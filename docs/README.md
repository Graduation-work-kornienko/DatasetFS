# Документация DatasetFS

Папка `docs/` — проектная вики. Источники истины для всего, что **не выводится из кода**: дизайн-решения, цели, методология бенчмарков, история оптимизаций.

## Если первый раз в проекте

1. **[../HANDOFF.md](../HANDOFF.md)** — быстрый bootstrap для новой сессии (2 мин), пишется по горячим следам.
2. **[architecture.md](architecture.md)** — как устроен DatasetFS целиком: компоненты, data-flow, IPC, конвенции, подводные камни.

## Если работаем дальше

3. **[status.md](status.md)** — где мы сейчас по фазам, что в работе, что в плане. Обновлять при значимых переходах.
4. **[optimizations/](optimizations/)** — журнал оптимизаций: одна папка = одна попытка. Описывает: контекст, baseline, гипотезу, реализацию, измерения, интерпретацию, следующий шаг. См. [optimizations/README.md](optimizations/README.md) для формата.

## Если разбираемся с тестами / бенчмарками

5. **[testing.md](testing.md)** — каталог тестов: что покрыто, что в gaps'е, как запускать.
6. **[benchmarking.md](benchmarking.md)** — каталог бенчмарков, метрики, текущие результаты, цели и недостающие метрики.

## Принципы документации

- **Не дублируем то, что выводится из кода.** Структура модуля, API сигнатуры, чему равны константы — есть в коде, читается напрямую. Здесь — **зачем** и **почему**.
- **Пишем по горячим следам.** Если решение принимается сейчас — фиксируется сейчас. Через месяц забудется, какие альтернативы рассматривались.
- **Ссылки относительные.** Все ссылки на код/файлы — относительными путями от файла. Кликабельно в IDE, не ломается при перемещении репозитория.
- **Русский язык.** Проект — дипломная работа, тексты должны быть переиспользуемы для thesis. Технические термины (sps, p50, RGB uint8 и т.п.) — без перевода.
